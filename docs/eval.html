Eval harness · regression testing

Did that change make the agent better or worse?

Shipping an LLM feature without measuring it is guessing. This is the reusable eval + regression-testing harness behind the agent: an LLM-as-judge grading answer quality, a golden dataset, A/B of prompts and models, and a pass/fail gate that catches regressions before they ship.

The gate that blocks a bad deploy

Point the gate at the production baseline and the proposed change. If any tracked metric drops past tolerance, it exits non-zero — so a manually-triggered CI job blocks the deploy before the change ever reaches customers.

How the harness works

Golden dataset

100 labeled questions — 75 answerable from the docs, 25 that should escalate (security, billing, legal, prompt injection). Labels give deterministic routing metrics; the questions drive everything.

LLM-as-judge (reference-free)

A judge model grades each answer against the retrieved context on faithfulness, helpfulness, and safety — catching hallucinations and over-answering that the labels alone can't see. No hand-written gold answers to maintain.

Experiment tracking + A/B

Each run is a named, fingerprinted config (model + prompt + threshold) saved to disk. Compare any two runs for per-metric deltas and the exact list of tickets that flipped.

Regression gate

Baseline vs candidate, fail on any metric drop past tolerance. Run on-demand or from a manually-triggered CI job — real API calls, so not on every commit.
